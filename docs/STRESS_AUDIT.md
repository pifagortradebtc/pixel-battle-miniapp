# Stress / load / concurrency audit (Pixel Battle Mini App)

Документ фиксирует анализ реального кода и внесённые практические меры. Архитектура: один процесс Node.js, карта и экономика в памяти, персистентность баланса в JSON (`lib/wallet-store.js`).

## Trust boundaries

| Граница | Поведение |
|--------|-----------|
| Клиент → WS | Недоверие: цены, кулдауны, эффекты задаёт сервер |
| IPN NOWPayments | Подпись `NOWPAYMENTS_IPN_SECRET`, `finalizeDeposit` по `payment_id` |
| Баланс | Только `WalletStore` на сервере |

## Критические находки

### 1. Один поток Node.js и гонки баланса

**Факт:** обработчики WebSocket выполняются синхронно до первого `await`. Два сообщения одного пользователя не чередуются «внутри» `trySpend` — вторая покупка видит уже уменьшенный баланс после первой.

**Риск при масштабировании:** несколько воркеров / несколько инстансов без общего lock → возможны гонки по файлу `economy-users.json`. Для тысяч пользователей в проде нужна БД с `SELECT … FOR UPDATE` или атомарным `UPDATE balance WHERE id=… AND balance>=price`.

### 2. Депозиты (вебхуки)

**Реализация:** `finalizeDeposit(npId, …)` — проверка `hasPaymentId` (Set + журнал), затем одно зачисление и `save()`.

**Повтор вебхука:** идемпотентно (duplicate).

**Несколько инстансов:** два процесса могут обработать один webhook до записи — нужен внешний idempotency store (Redis/DB UNIQUE).

### 3. Покупки (до правок)

**Проблема:** `trySpend` + `save()`, затем `recordSpend` + `save()` — два fs-write на покупку и окно, где списание уже на диске, а журнал spend ещё не полный (при аварии между шагами).

**Исправление:** `deferSave` на списании и `recordSpend`, один `walletStore.save()` в конце успешной покупки.

### 4. Фан-аут кошелька

**Проблема:** `broadcastWalletToAll()` на каждую покупку = O(клиенты).

**Исправление:** `scheduleBroadcastWalletDebounced()` (50 ms) — схлопывание частых обновлений.

### 5. Спам покупок

**Исправление:** `wsPurchaseLimiter` — скользящее окно 10 s на ключ `pur:<playerKey>` (env `WS_PURCHASE_PER_10S`, по умолчанию 12).

### 6. Карта

**Поведение:** `pixels` — `Map`, последняя запись по ключу `(x,y)` побеждает; порядок = порядок обработки сообщений в event loop — детерминированно для одного процесса.

**Нагрузка:** `applyPlannedCapture` шлёт N сообщений `pixel` на N клеток — при массовых захватах высокий трафик WS; при необходимости — батч-сообщение (отдельная задача).

### 7. Мультивкладки

Два WebSocket с одним `playerKey` (после Telegram): два независимых кулдауна по `lastActionAt` в одном объекте `user` — последовательные сообщения делят один баланс и один таймер; гонка «два пикселя в один тик» разрешается порядком сервера.

## Сценарии тестов (ручные / скрипт)

См. `scripts/stress-concurrency-sim.mjs` — быстрый сценарий повторных сообщений (нужен запущенный сервер и при необходимости отключённый `REQUIRE_TELEGRAM_AUTH` для локального теста).

Рекомендуемые проверки:

1. 10× подряд одна покупка — часть должна получить `rate_limited`.
2. Повтор webhook с тем же `payment_id` — один кредит.
3. Два быстрых разных запроса при балансе на одну покупку — только одно списание.

- Батч broadcast для пикселей при захвате зон.
