# Эксплуатация в продакшене

**Для владельца без опыта в коде** — см. [HOSTING-DLYA-VAS.md](HOSTING-DLYA-VAS.md). Шаблон переменных — файл `env.example` в корне репозитория.

Кратко: как поднять несколько инстансов, Redis, восстановление карты после рестарта и переменные окружения.

## Масштабирование (несколько Node-процессов)

- Задайте **`REDIS_URL`** — общий Pub/Sub для игровых сообщений (те же JSON, что уходят в WebSocket).
- На **одном** инстансе выставьте **`CLUSTER_LEADER=true`** (или не задавайте `CLUSTER_LEADER` — по умолчанию процесс считается лидером). На остальных — **`CLUSTER_LEADER=false`**.
- Таймеры раунда, Telegram long poll, тяжёлые «ведущие» действия выполняются только на лидере. Вторичные инстансы принимают WebSocket от игроков и ретранслируют состояние с Redis.

## Снимок карты (`data/pixels-snapshot.json`)

- Пишется **только на лидере** (или в одиночном процессе без Redis). Вторичные ноды файл не загружают и не перезаписывают — синхронизация через Redis/WS.
- После старта: после `rebuildLandFromRound` и **до** восстановления спавнов загружается снимок, если совпадают `roundIndex`, `gridW`, `gridH`.
- Сохранение: debounce (**`PIXELS_SNAPSHOT_DEBOUNCE_MS`**, по умолчанию 3000) и периодически (**`PIXELS_SNAPSHOT_INTERVAL_MS`**, по умолчанию 45000). При **`SIGTERM`/`SIGINT`** выполняется синхронный сброс снимка и сброс очереди batched-пикселей.
- Отключить снимок: **`PIXELS_SNAPSHOT_DISABLE=true`**.

## Сеть и нагрузка

- Исходящие изменения клеток с сервера группируются в сообщения **`pixelBatch`** (формат `v2`: `[x, y, teamId, ownerPlayerKey, shieldedUntil]`), чтобы снизить число кадров WS/PubSub при массовой окраске. Размер одного пакета ограничен (**`PIXEL_BATCH_MAX_CELLS`**, по умолчанию 800, макс. 4000).
- Клиент ограничивает размер входящего JSON (**`MAX_WS_INCOMING_CHARS`** в `js/main.js`).

## Рекомендуемая инфраструктура для высокой нагрузки

- Отдельный **Redis** (managed или собственный) с низкой задержкой до приложений.
- Несколько **веб-инстансов** за балансировщиком с **sticky sessions** по WebSocket (или корректная поддержка WS в L7-балансировщике). Игровые клики обрабатывает **только лидер**; вторичные инстансы получают те же события из Redis и отдают их своим подключённым клиентам (без повторной рассылки на лидере — дубли отключены).
- Лимиты: `ulimit`/память под число одновременных WS, таймауты прокси выше интервала пинга.
- Дальнейшее масштабирование (отдельный game-сервис, бинарный протокол, дельты по viewport) — по отдельной дорожной карте.
